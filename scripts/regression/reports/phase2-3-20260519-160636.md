### US-015 (production security headers) — **NOT directly testable in dev**

The new `SECURE_HSTS_SECONDS`, `SECURE_HSTS_INCLUDE_SUBDOMAINS`, `SECURE_HSTS_PRELOAD`, `SECURE_CONTENT_TYPE_NOSNIFF`, `SECURE_REFERRER_POLICY` are all gated on `if not DEBUG:` in `bibliotype/settings.py`. Local dev runs with `DEBUG=True`, so these headers are absent here by design. They'll fire in prod where `DEBUG=False` is enforced by Phase 1's `ImproperlyConfigured` startup assert. **Verify on prod after deploy:**

```bash

#   x-content-type-options: nosniff
#   referrer-policy: same-origin
```

## Observations (non-blocking)

### O1. T15 case (c): row cap + lenient schema gap

The view-level US-018 check correctly truncates 60 000 → 50 000 rows and dispatches. My synthetic CSV had `Title,Author,Date Read,Number of Pages,Average Rating` — enough to pass the `{"Title","Author"} ⊆ columns` check but the deeper `dna_analyser.calculate_full_dna` rejected with `Unrecognized CSV format. Please upload a Goodreads or StoryGraph export.` The task FAILED cleanly with that message in the user-facing flow.

This is **layered defense working**, not a finding:
- View caps memory before any task work (US-018 goal)
- Downstream service performs deeper detection and fails fast with a clear error
- No resources wasted on enrichment

But the PR description's "US-018 no defense-in-depth at the task layer" caveat is real: `generate_reading_dna_task` itself does not re-validate row count or columns. If a future internal caller (management command, async backfill) dispatches the task with an oversize payload, it bypasses the view cap. Worth a 5-line `assert len(rows) <= MAX_UPLOAD_ROWS` in the task at some point.

### O2. US-016 rate-limit is keyed on REMOTE_ADDR — verify Nginx forwards client IP on prod

Already called out as "Known limitation #1" in the PR description. Confirmed at the code level (`@ratelimit(key="ip", rate="5/m", method="POST", block=True)`). Locally, since all requests come from the host machine through Docker, the per-IP rate-limit acted as a global bucket during my testing — which is exactly what would happen in prod if Nginx isn't forwarding the real client IP. **Pre-merge action**: SSH to the VPS and confirm `proxy_set_header X-Real-IP $remote_addr;` (or equivalent) is set in the Nginx `location / { }` block, and that Django reads it (consider `django-ipware`). Otherwise US-016 becomes "site-wide 5/m" which would let a single attacker DoS all login attempts globally.

### O3. US-017 still leaks via `Location` header

The duplicate-email branch (`302 → /password-reset/done/`) and the fresh-email branch (`302 → /`) have different `Location` headers and different `Set-Cookie` payloads (only the fresh-email branch sets `sessionid`). An attacker scripting signup attempts can still tell them apart. PR description acknowledges this. Substantial reduction from the prior explicit "Email already taken" error, but not "observationally identical".

### O4. dna_analyser `ThreadPoolExecutor` change (uncommitted at test time)

The worker change from `max_workers=1` to `max_workers=8` was active during this run. T1 / T1b / T3 / T8 all generated DNA without database lock errors. Worker logs show no `IntegrityError` or `OperationalError`. Postgres row-level locking handles the contention as advertised. ✓ (Note: this change is not in any commit yet — if it's intended for PR #107, it should be `git add`ed before merge; if not, it should be stashed.)

### O5. Pre-existing background-enrichment quirk

Worker logged `Author Status Task Error: Author with ID 63 not found` — a background enrichment task fired against an Author row that's been deleted (probably from one of my earlier test runs that scrubbed the local DB). Pre-existing, unrelated to this PR.
