## What's still broken (F2 only)

`@ratelimit(key=client_ip_key, rate="5/m")` on `_login_view_throttled` is in place in the code, but `HTTP_X_REAL_IP` and `HTTP_X_FORWARDED_FOR` aren't reaching Django in a usable form. Nginx config almost certainly needs:

```nginx
proxy_set_header X-Real-IP $http_cf_connecting_ip;
proxy_set_header X-Forwarded-For $http_cf_connecting_ip;
```

Then `nginx -s reload`. No app rebuild.

## Why F1 looked worse than it was

Both runs of T1/T1b submitted while the worker was still warming up post-deploy. The deploy finished at 09:39:38 UTC; first uploads at 09:49 UTC. Possible causes for the long initial latency:
- Worker cold-start (Python process loading models, opening DB pool)
- A bunch of background tasks queued from the deploy startup (signals, periodic chord registration etc.) consuming the worker thread
- Postgres cold cache for `Book`/`Author` lookups on synthetic libraries
- External API throttling (Gemini quota on a fresh day cycle)

By the time the re-run submitted at 10:11, the worker had been processing for 30+ min and the next upload completed in 4 min. Still slower than the 2026-05-15 prod baseline (SUCCESS on first poll, <30s) but not catastrophic.

**Recommendation**: monitor for 24 h. If new uploads consistently take 3–5 min, that's a regression worth tracking down (probably the missing `ThreadPoolExecutor max_workers=8` working-tree change from the May 19 local test never made it into PR #107). If it settles back to <1 min, it was just deploy-startup overhead.
