## Findings

### F1 (P0) — Celery worker not producing task results

**Symptom**: Two independent anonymous uploads (synthetic_lit_fiction1.csv with 23 books, storygraph_small.csv with 12 books) submitted at 09:49 and 09:54 UTC, both still in `{"status": "PENDING"}` at 10:03 UTC. ~14 min and ~9 min respectively.

**What we know**:
- The upload POST succeeded (HTTP 302 → `/task/<id>/`) → view-level dispatch worked
- The `task_owner_<id>` Redis binding was created → confirmed via T7 returning 403 on cross-session GET
- Therefore: Celery broker is reachable, Redis is healthy, the *dispatch* path works
- What's broken: the worker side — either no worker is consuming from the queue, or the worker is consuming but never producing a result

**Possible causes (in likelihood order)**:
1. **Worker container crashed during deploy and didn't recover.** Phase 1's F1 (worker missing `DJANGO_ENV` in compose) was supposedly fixed — but if the deploy used an older compose definition, or if a different env var is now missing post-Phase-3, the worker would crash-loop. The brief 502 window (90s) could mask a worker failure that doesn't take down the web service.
2. **Gemini API quota exhausted.** The DNA pipeline calls Gemini for the "vibe" generation. If quota is hit, the task may retry indefinitely rather than fail fast. Previous prod tests on May 15 ran in early UK morning when quota was fresh.
3. **Worker pool starvation.** The uncommitted `ThreadPoolExecutor max_workers=8` change from the local 2026-05-19 test was *not* in PR #107 (still working-tree-only). Prod still runs `max_workers=1` for inner-loop concurrency. But even so, the OUTER Celery task should complete eventually — it's just slower.
4. **Worker bind mount / image issue.** If the deployed image is missing a dependency or has a broken entrypoint, the worker container might be in a restart loop visible only via `docker ps`.

**Recommended diagnostic** (on the VPS):
```bash
ssh prod
docker compose -f docker-compose.prod.yml ps                  # is worker Up?
docker compose -f docker-compose.prod.yml logs --tail=200 worker
docker compose -f docker-compose.prod.yml logs --tail=200 worker | grep -iE 'error|exception|traceback|gemini|quota|429|503'
docker compose -f docker-compose.prod.yml exec redis redis-cli -n 0 LLEN celery   # backlog?
```

**User impact**: Anyone uploading a CSV right now sees a "processing…" screen and never gets their Bibliotype. This is the primary user flow.

### F2 (P1) — US-016 login rate-limit not enforcing on prod

**Symptom**: 19 consecutive wrong-password POSTs from my IP all returned HTTP 200 + "Invalid email or password". The expected behavior is HTTP 429 starting on the 6th attempt within a 60s window. Locally on 2026-05-19, attempts 1–5 returned 200 and 6+ returned 429 — exact behavior.

**Code is correct** (`core/views.py:1021` + `core/ratelimit_utils.py`):
```python
@ratelimit(key=client_ip_key, rate="5/m", method="POST", block=True)
def _login_view_throttled(request): ...

def client_ip_key(_group, request):
    return get_real_client_ip(request) or "unknown"

def get_real_client_ip(request):
    real_ip = request.META.get("HTTP_X_REAL_IP")
    if real_ip: return real_ip
    client_ip, _ = get_client_ip(request)  # ipware
    return client_ip
```

**Likely root cause**: `HTTP_X_REAL_IP` and `HTTP_X_FORWARDED_FOR` are not being set by Nginx on prod (or are being set to a value that's varying per request). The PR description called this out as "Known limitation #1" pre-merge:

> US-016 IP keying behind Nginx. `@ratelimit(key="ip")` reads `REMOTE_ADDR`. If Nginx isn't explicitly forwarding the client IP via `proxy_set_header` AND a middleware like `django-ipware` isn't installed, `REMOTE_ADDR` will be the proxy's IP and the 5/m limit becomes site-global instead of per-client.

The empirical behavior I'm seeing (19 attempts, no 429) is **worse than site-global**: even a global 5/m bucket should 429 after 5 hits. Two sub-hypotheses for why it's not firing at all:

(a) **`client_ip_key` returns `"unknown"` on every request AND something in django-ratelimit's flow short-circuits.** Unlikely — `"unknown"` should still act as a constant bucket. Worth verifying.
(b) **Some upstream layer (Cloudflare cache, or an Nginx config) is producing different IPs per request to Django.** Could be `X-Forwarded-For` chains where Cloudflare prepends multiple IPs and ipware picks a different one per request.

**Recommended diagnostic** (on the VPS):
```bash
# 1. Confirm Nginx is forwarding headers
ssh prod
cat /etc/nginx/sites-enabled/bibliotype.conf | grep -iE 'proxy_set_header|real_ip'
# Expect to see:
#   proxy_set_header X-Real-IP $remote_addr;
#   proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
# Or with cloudflare's CF-Connecting-IP:
#   proxy_set_header X-Real-IP $http_cf_connecting_ip;

# 2. Live-trace what Django sees:
docker compose -f docker-compose.prod.yml exec web poetry run python manage.py shell
>>> # add a temporary middleware that logs request.META keys
```

**Fix**: Update Nginx config to forward Cloudflare's `CF-Connecting-IP` as `X-Real-IP`. The Django code is already wired to read `HTTP_X_REAL_IP` first via `get_real_client_ip()`.

**Impact**: Login is currently unrate-limited on prod. A credential-stuffing attacker can attempt unlimited passwords per second. The other rate-limit (US-020, username API) is keyed on `user`, which works regardless of IP — so the privacy/profile abuse surface is still covered.

## Phone-friendly summary

The deploy went out but two things are broken on prod that weren't broken in the local test of #107. **CSV uploads aren't completing** — submitted at 09:49 UTC, still "processing" 14+ minutes later, which means the Celery worker is either dead or stuck. SSH in and run `docker compose -f docker-compose.prod.yml logs --tail=200 worker` first — if the worker is in a crash loop, `docker compose restart worker` will probably bring it back. Second issue is **the login rate-limit (US-016) isn't actually rate-limiting** — I submitted 19 wrong passwords from my IP and got 200 every time. The Django code is correct; the problem is almost certainly that Nginx isn't forwarding the client IP to Django, exactly as the PR's "Known limitation #1" warned. Fix is `proxy_set_header X-Real-IP $http_cf_connecting_ip;` in the Nginx config + `nginx -s reload`. The rest of the deploy is fine: HSTS/nosniff/referrer-policy are live, CSV schema cap rejects bad files at the HTTP layer, 404 page is the custom one, cross-session 403 works, static assets serve. **Verdict: NOT_READY for normal traffic until F1 is resolved.** F2 is a security regression but doesn't make the site unusable.
