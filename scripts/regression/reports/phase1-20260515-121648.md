## Findings

### F1. **REGRESSION** ‚Äî Celery worker can't restart in local dev after Phase 1 ‚Äî `docker-compose.local.yml` worker service is missing `DJANGO_ENV`
- **Severity:** High (blocks any worker restart; new deploy = silent Celery outage in dev)
- **Phase 1 commit responsible:** `e978af4 feat: [US-004] - DEBUG defaults to False and add startup assert in production`
- **Root cause:** US-004 added an `ImproperlyConfigured` guard that refuses to boot when `DEBUG=True` and `DJANGO_ENV` is not in `{"development", "test", "ci"}`. The `web` service in `docker-compose.local.yml:14` sets `DJANGO_ENV=development`. The `worker` service (around line 47‚Äì67) does **not**. Existing worker containers survived because they were started from an older image; on restart, they crash-loop with:
  ```
  django.core.exceptions.ImproperlyConfigured: DEBUG must be False when DJANGO_ENV=''.
  Permitted DEBUG=True environments: development, test, ci.
  ```
- **Fix:** add `- DJANGO_ENV=development` to the worker service env block in `docker-compose.local.yml`. (Also worth auditing `docker-compose.prod.yml` and any CI compose files.)

### F2. **REGRESSION** ‚Äî Worker stale-code hazard for the claim flow
- **Severity:** High (data-loss-on-deploy for users mid-signup)
- **Phase 1 commit responsible:** `ff16c64 feat: [US-003] - Bind task_id_to_claim on signup to caller's pre-login session_key`
- **Root cause:** US-003 changed `claim_anonymous_dna_task`'s signature from 2 args to 3 args (and the caller now passes the pre-login session_key). Celery workers don't autoreload; if web pods deploy the new code but workers are not restarted, every signup-with-claim emits `TypeError('claim_anonymous_dna_task() takes 3 positional arguments but 4 were given')`, leaving the user account with no DNA and `pending_dna_task_id` stuck. The `signup_view` does not fall back to `request.session["dna_data"]` when `task_id_to_claim` is set, so the data is permanently lost.
- **Fix in deploy procedure:** ensure worker is restarted after merge. Or, for defense-in-depth, accept variable kwargs in the task and log warnings for signature drift. The PR's commit message should call this out as a deployment ordering requirement.
- **Note:** Once the worker runs the latest code, the new path is correct (verified by replaying `claim_anonymous_dna_task.apply(...)` via Django shell ‚Äî the user's profile populated immediately and the dashboard rendered fully).

### F3. **MINOR** ‚Äî Test prompt has stale expectations
- T2 expects redirect to `/dashboard/` after signup-without-claim; the code redirects to `core:home` (`/`). Not a bug.
- T10 expects the custom 404 to render in dev; Django's `DEBUG=True` always pre-empts `handler404`. Not a bug.
- T7 expects warn-and-allow when `ENFORCE_TASK_OWNERSHIP=False`; the code now always 403s. This is a *stronger* posture than the test prompt anticipates.
- `csv/` directory (not `core/tests/fixtures/csv/`) is where fixtures live; the test prompt's first path is dead.
- Recommend a quick edit pass on `scripts/regression/phase1-chrome-mcp-prompt.md` once F1/F2 are fixed.

### F4. **OBSERVATION** ‚Äî `ENFORCE_TASK_OWNERSHIP` is now dead code in the request path
- The flag is still read in `bibliotype/settings.py:51` but no code branches on it. Either remove it (cleanest) or keep it but update the docstring to say "retained for deploy-doc compatibility; no functional effect". The comment in `core/views.py:1246` already acknowledges this, but `.env.example` and any runbooks may still reference it.

### F5. **OBSERVATION** ‚Äî Worker logs contain Google Books 503s and Gemini 429 rate-limit errors
- Unrelated to Phase 1. Pre-existing externally-driven rate limiting. Worth tracking but out of scope.
