# Bibliotype triage progress log

## Codebase Patterns
- Anonymous DNA tasks: bind `task_owner_<task_id>` in cache to
  `request.session.session_key` at upload time (US-001). Session must be
  saved first if `session_key is None` (Django doesn't allocate one until
  first save). Reads of `task_owner_*` happen in `get_task_result_view`
  (US-002) and `claim_anonymous_dna_task` (US-003).
- Test environment: `black` / `isort` are NOT installed in the Docker image
  or the local Poetry venv (not in `pyproject.toml [tool.poetry.group.dev]`,
  which only has `djlint`). Match existing style by hand and skip the
  formatter step until they're added as a dev dep.
- Class-to-module-functions refactor (US-031a recipe): replace
  `class Foo: __init__` defaults with module constants; each
  `def method(self, ...)` becomes `def method(...)`; `self.attr` →
  `MODULE_CONSTANT`; `self.helper()` → `helper()`. `@patch` paths drop the
  `.ClassName` segment: `pkg.mod.ClassName.method` → `pkg.mod.method`. If
  the class had a method AND a same-named convenience function that wrapped
  it with extra behavior (e.g. caching), keep both layers — public name
  stays on the wrapper; pipeline gets an `_uncached` (or similar) underscore
  prefix.

## 2026-05-14 16:35 — US-005
- Carved `ADMIN_COMMANDS` + `ALLOWED_COMMANDS` out of `core/admin.py` into a new
  `core/management_command_registry.py` module (single source of truth).
  `core/admin.py` now imports from it (the `ALLOWED_COMMANDS` symbol is the
  same `frozenset` object), and `core/tasks.py` imports it at module top and
  raises `ValueError(f"command not allowed: {command_name}")` as the first
  statement of `run_management_command_task`. This makes the worker reject
  unwhitelisted names even when a caller publishes directly to the broker,
  bypassing the existing HTTP-layer check in `command_run_api`.
- Files changed:
  - core/management_command_registry.py (new — module-level list + frozenset)
  - core/admin.py (deleted inline `ADMIN_COMMANDS`/`ALLOWED_COMMANDS`,
    imports them from the new module)
  - core/tasks.py (`from .management_command_registry import ALLOWED_COMMANDS`
    + first-line whitelist guard in `run_management_command_task`)
  - core/tests/test_tasks_unit.py (new `RunManagementCommandTaskWhitelistTests`
    class: 1 negative test that `call_command` is not invoked on a blocked
    name, 1 empty-string negative test, 1 positive test that a whitelisted
    name reaches `call_command`).
  - core/tests/test_integration.py (two pre-existing tests
    `test_run_management_command_task_captures_output` and
    `..._captures_error` passed arbitrary names `"test_command"` /
    `"failing_command"` and now hit the new whitelist guard; switched both
    to the whitelisted no-op `"analyze_genres"`. Behaviour they assert —
    stdout capture, CommandError wrapping — is unchanged.)
  - scripts/ralph/prd.json (US-005 passes=true)
- Learnings for future iterations:
  - **Don't import from `core.admin` in `core/tasks.py`.** `core/admin.py`
    already does a deferred `from .tasks import run_management_command_task`
    inside `command_run_api`. Lifting `ALLOWED_COMMANDS` to module-top of
    `tasks.py` and importing it from `admin` would invert the dependency at
    Django-startup time when the admin module loads. The new
    `core/management_command_registry.py` module sits below both and is
    imported by both — no cycle.
  - **`@shared_task` tasks are callable as plain Python via `.run(...)`.**
    For pure-logic unit tests (no Celery routing, no broker) prefer
    `task.run(arg)` over `task.apply()` / `task.delay()` — it bypasses the
    EagerResult wrapping so `assertRaises(ValueError)` works without
    `throw=True` / `propagate=True` plumbing. Use `.delay()` only when the
    test specifically needs the Celery dispatcher (e.g. mocking it to
    assert "was this enqueued?").
  - **The `call_command` import inside `run_management_command_task` is
    intentional.** Patching `core.tasks.call_command` from a test would NOT
    work because the name isn't bound at module scope; patch
    `django.core.management.call_command` instead, OR pin the import to
    a known module path. Existing tests use the latter.
  - The HTTP layer (`command_run_api`) still validates `command_name not in
    ALLOWED_COMMANDS` and returns a 400. The task-level guard is the
    defence-in-depth backstop, not a replacement.
- Tests in scope passed: `core.tests.test_tasks_unit core.tests.test_tasks_integration
  core.tests.test_integration` → 67 / 67 green. Full Django suite also exit 0.

  - **`@ratelimit(..., block=True)` raises `Ratelimited` BEFORE the view body

    essential.

  - `cache.add()` is atomic in Django's cache framework — returns True only
    when the key was newly set. The `safe_cache_*` family was missing an
    `add` wrapper; future stories that need atomic set-if-absent semantics
    (e.g. distributed locks, dispatch guards) should use `safe_cache_add()`.
    Fail-open on Redis errors matches the rest of the safe wrappers and
    keeps task dispatch from getting stuck when the cache is unavailable.
  - Celery's `self.retry(...)` raises `Retry`, which propagates through
    `try/except Exception`. A naive `finally` block would clear the sentinel
    on retry too, breaking the guard during the retry countdown. The fix is
    a sticky flag set to `False` in the retry-bound except branch BEFORE
    the `raise self.retry(...)`, then checked in `finally`.
  - The `generate_recommendations_task` lazy-imports `safe_cache_delete`
    inside the function body. Moving it to the top of the function (alongside
    the new `safe_cache_delete(f"recs_dispatching_{user_id}")` call) avoids
    a second import in `finally` and keeps both delete sites consistent.
  - Test pattern for `mock.patch` on Celery tasks that the SUT imports lazily:
    patch the canonical reference (`core.tasks.generate_recommendations_task.delay`),
    not the local import inside the view — the view's lazy import binds to
    the patched object at call time.

    `LocMemCache` persists state across test methods within the same
    TestCase (it's process-scoped, not request-scoped); without it the
    sentinel from a prior test could leak in.

  - scripts/ralph/prd.json (US-025 passes=true)
- Learnings for future iterations:
  - **`order_by("?")` on Postgres compiles to `ORDER BY random()`** which is
    a full-table scan + sort. For tables of any meaningful size this is
    O(N log N) on every cache miss. Sampling ids first is O(N) for the id
    fetch + O(K) for `random.sample`, then an `IN`-clause SELECT which uses
    the primary-key index. Win is biggest as N grows.
  - **`random.sample` raises ValueError when sample size > population**.
    The `min(len(ids), 100)` guard is essential — without it, a fresh DB
    with <100 anonymized profiles would 500 on every recs cache miss.
  - **`values_list("id", flat=True)` on a model with PK `id` is a single
    column index-only scan in Postgres** (cheaper than `values_list("pk")`
    which sometimes triggers a SELECT *). Stick with `"id"` literal.
  - The cached value (`anon_profiles_sample`) is a list of model instances
    pickled into Redis. Switching the sampling strategy keeps the shape
    identical; the only observable difference is that on cache miss the
    100 instances come back via `filter(id__in=...)` in arbitrary PK order
    rather than `random()` order, but downstream code iterates the list
    without ordering assumptions.
